# Этап 07. Полноценное учебное fullstack-приложение

## Учебная цель

Собрать цельное учебное приложение:

- backend на Go;
- frontend на HTML/CSS/JS без фреймворков;
- SQLite;
- создание, редактирование, удаление;
- фильтрация и поиск;
- наглядные логи и понятная структура.

Поток данных:

`пользователь -> форма/кнопка -> JS -> fetch -> handler -> SQL -> SQLite -> JSON -> JS -> DOM`

## Как запустить

Из каталога этапа:

```bash
go run .
```

Из корня репозитория:

```bash
make run-07
```

По умолчанию приложение слушает порт `18087`.

## Какие URL открыть

- `http://localhost:18087/` — frontend
- `http://localhost:18087/api/tasks` — JSON API

## Что умеет приложение

- показать список задач;
- создать задачу;
- отредактировать задачу;
- удалить задачу;
- фильтровать по статусу;
- фильтровать по категории;
- искать по заголовку и описанию;
- показывать статус, категорию и дату создания.

При первом запуске сервер сам создает файл SQLite и добавляет несколько
демонстрационных задач, чтобы UI сразу был наполнен данными.

Backend валидирует:

- JSON-тело запроса;
- допустимые значения `status` и `category`;
- query-параметры фильтрации.

## Архитектурная схема

```text
Браузер
  -> frontend/index.html
  -> frontend/app.js
  -> fetch("/api/tasks")
Go HTTP server
  -> общий HTTP-лог
  -> handler API
  -> store
  -> SQLite
  -> JSON-ответ
JavaScript в браузере
  -> обновление DOM
```

## Структура каталогов


## Как проходят данные

### Создание задачи

1. Пользователь заполняет форму.
2. `frontend/app.js` собирает данные в объект JavaScript.
3. `fetch` отправляет `POST /api/tasks` с JSON.
4. Handler в Go валидирует входные данные.
5. Store выполняет `INSERT` в SQLite.
6. Backend возвращает JSON созданной задачи.
7. Frontend заново запрашивает список и перерисовывает DOM.

### Редактирование задачи

1. Пользователь нажимает `Редактировать`.
2. JS заполняет форму данными выбранной задачи.
3. После отправки идет `PUT /api/tasks/{id}`.
4. Handler вызывает `UPDATE` в SQLite.
5. UI получает обновленные данные и показывает их в списке.

### Фильтрация и поиск

1. Пользователь вводит строку поиска и выбирает фильтры.
2. Frontend вызывает `GET /api/tasks?q=...&status=...&category=...`.
3. Backend собирает SQL `SELECT` с условиями.
4. SQLite возвращает только подходящие записи.
5. Frontend отображает отфильтрованный список.

Если передать недопустимый `status` или `category`, backend вернет `400 Bad Request`
с понятным JSON-описанием ошибки. Это удобно показывать как пример явной
валидации query-параметров.

## Пошаговый сценарий демонстрации на лекции

1. Открыть UI и показать список задач.
2. Создать новую задачу и параллельно открыть вкладку `Network`.
3. Показать `POST /api/tasks` и JSON-тело запроса.
4. Перейти в Go-код handler и затем в SQL-вставку.
5. Вернуться в браузер и показать, как DOM обновился.
6. Выбрать задачу на редактирование.
7. Показать `PUT /api/tasks/{id}`.
8. Продемонстрировать фильтр по статусу и поиск по тексту.
9. Удалить запись и показать `DELETE`.
10. Обратить внимание на общий HTTP-лог в терминале.

## Примеры curl

Получить список:

```bash
curl -i http://localhost:18087/api/tasks
```

Создать задачу:

```bash
curl -i -X POST http://localhost:18087/api/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title":"Подготовить финальное демо",
    "description":"Показать путь от fetch до SQLite",
    "status":"todo",
    "category":"lecture"
  }'
```

Обновить задачу:

```bash
curl -i -X PUT http://localhost:18087/api/tasks/1 \
  -H "Content-Type: application/json" \
  -d '{
    "title":"Подготовить финальное демо",
    "description":"Теперь со сценарием отладки",
    "status":"doing",
    "category":"demo"
  }'
```

Удалить задачу:

```bash
curl -i -X DELETE http://localhost:18087/api/tasks/1
```

Фильтрация:

```bash
curl -i "http://localhost:18087/api/tasks?status=todo&category=lecture&q=demo"
```

## Где удобно ставить breakpoint

- вход в handler: `backend/api.go`
- парсинг JSON: `decodeTaskInput`
- слой работы с БД: `Store.Create`, `Store.Update`, `Store.List`
- генерация HTTP-ответа: `writeJSON`
- frontend fetch и обработка ответа: `frontend/app.js`, функции `saveTask`, `loadTasks`, `deleteTask`

## Что логировать при отладке

- метод и URL запроса;
- фильтры `q`, `status`, `category`;
- входной JSON;
- SQL-операцию и ее результат;
- HTTP status и длительность запроса.

## Простая конфигурация

По умолчанию используются:

- порт `:18087`;
- база `data/tasks.db`.

При необходимости их можно переопределить через переменные окружения:

- `APP_PORT`
- `APP_DB_PATH`

## Архитектурное решение

Здесь структура уже чуть богаче, чем на ранних этапах, но она по-прежнему учебная:

- отдельный `backend/` нужен, чтобы показать разделение ролей;
- store работает через `database/sql` и явные SQL-запросы;
- HTTP-логирование оставлено в виде одной простой обертки вокруг `ServeMux`;
- frontend остается на чистом JavaScript без сборки и без фреймворков.

Это дает достаточно реалистичный fullstack-поток, но не превращает пример в тяжелую продакшен-архитектуру.
