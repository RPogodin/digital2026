# Этап 05. CRUD с SQLite

## Учебная цель

Развить JSON API из прошлого этапа и заменить хранение в памяти на постоянное
хранение в SQLite.

Поток данных:

`клиент -> HTTP-запрос -> handler -> SQL -> SQLite -> JSON-ответ`

## Как запустить

Из каталога этапа:

```bash
go run .
```

Из корня репозитория:

```bash
make run-05
```

## Какие URL использовать

- `http://localhost:18085/api/items`
- `http://localhost:18085/api/items/1`

## Что меняется по сравнению с in-memory

На этапе 04 данные жили только в памяти процесса.
После перезапуска сервера все пропадало.

На этом этапе:

- данные пишутся в файл SQLite;
- база создается автоматически при старте, если файла еще нет;
- если таблица пуста, сервер сам добавляет пару стартовых записей для демонстрации;
- после перезапуска сервера записи сохраняются.

Файл базы данных создается прямо в каталоге этапа:

```text
05_crud_with_sqlite/tasks.db
```

## Зачем нужна БД

База данных нужна, когда данные должны жить дольше одного запуска процесса.
Даже для маленького учебного примера это полезно, потому что сразу видно
разницу между временным и постоянным хранением.

## Где происходит SQL-запрос

SQL виден прямо в Go-коде:

- `listItems` делает `SELECT`;
- `getItem` делает `SELECT ... WHERE id = ?`;
- `createItem` делает `INSERT`;
- `updateItem` делает `UPDATE`;
- `deleteItem` делает `DELETE`.

Это намеренно сделано без ORM, чтобы путь данных оставался прозрачным.

## Какие HTTP status code показывает этап

- `200 OK` — успешное чтение и обновление;
- `201 Created` — успешное создание;
- `204 No Content` — успешное удаление;
- `400 Bad Request` — некорректный JSON, лишние поля или неверный `id`;
- `404 Not Found` — элемент не найден;
- `500 Internal Server Error` — ошибка работы с базой данных.

## Как проходят данные

1. Клиент отправляет HTTP-запрос.
2. Handler разбирает входные данные и валидирует их.
3. Дальше вызывается функция с SQL-запросом.
4. SQLite читает или изменяет данные в файле БД.
5. Результат снова превращается в Go-структуру.
6. Сервер отдает JSON-ответ.

## Примеры curl

Получить список:

```bash
curl -i http://localhost:18085/api/items
```

Создать элемент:

```bash
curl -i -X POST http://localhost:18085/api/items \
  -H "Content-Type: application/json" \
  -d '{"title":"Задача из SQLite","done":false}'
```

Обновить элемент:

```bash
curl -i -X PUT http://localhost:18085/api/items/1 \
  -H "Content-Type: application/json" \
  -d '{"title":"Обновленная запись","done":true}'
```

Удалить элемент:

```bash
curl -i -X DELETE http://localhost:18085/api/items/1
```

## Где ставить breakpoint

- В `handleCreate` и `handleUpdate` перед разбором JSON.
- В `createItem`, `updateItem`, `deleteItem` на строках с `db.Exec`.
- В `getItem` и `listItems` на строках с `db.QueryRow` и `db.Query`.
- В `writeJSON`.

## Что логировать при отладке

- входной JSON;
- `id` из URL;
- SQL-операцию, которая сейчас выполняется;
- количество затронутых строк;
- статус ответа.

## Архитектурное решение

Здесь оставлен прямой стиль:

- handler и работа с БД находятся рядом;
- SQL написан явно строками;
- отдельные функции нужны только для того, чтобы показать конкретный шаг обработки.

Это удобнее для обучения, чем прятать логику за тяжелым слоем абстракций.
